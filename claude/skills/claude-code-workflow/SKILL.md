# Claude Code Workflow

Patterns for getting the most out of Claude Code across any codebase.

## Read before you edit

Always read the file before editing it. Claude Code's Edit tool requires a prior Read in the same session — this is also just good practice.

```
Read → understand → Edit (targeted diff, not full rewrite)
```

## CLAUDE.md hierarchy

Claude Code loads CLAUDE.md files in this order (all merged):
1. `~/.claude/CLAUDE.md` — global, every session, every machine
2. `<repo-root>/CLAUDE.md` — project-specific overrides
3. Subdirectory CLAUDE.md files — scoped to that subtree

Use this to layer: global defaults → repo conventions → folder-level specifics.

## Parallel tool calls

When multiple reads or searches are independent, call them in parallel in one message — faster and keeps context clean:

```
✅ Read file A + Read file B (same message, no dependency)
❌ Read file A → wait → Read file B (sequential when not needed)
```

## Autonomous loop behavior

When running in `/loop` mode:
- Act on what the conversation already established — don't invent new work
- Reversible actions (edits, tests): make the call and proceed
- Irreversible actions (push, delete, send): wait for explicit authorization
- Three consecutive "nothing to do" results → scale back to a quick check and stop

## Memory system

Persistent memory lives in `~/.claude/projects/<path-hash>/memory/`.
Use it to carry context across sessions: user preferences, project decisions, feedback, references.
See `~/ai-agent-skills-sync/claude/skills/claude-memory-system/SKILL.md` for the full pattern.

## Subagents

Use the Agent tool for:
- Open-ended codebase exploration spanning many files (use `subagent_type: Explore`)
- Independent parallel research tasks
- Tasks that would pollute the main context with excessive output

Do not spawn subagents for tasks you can do directly in 1-3 tool calls.

## On-demand skill loading

When starting a task that clearly matches a domain, read the relevant skill file before writing code:

| Task involves | Read first |
|---|---|
| async/await, promises, queues | `~/ai-agent-skills-sync/skills/async-concurrency/SKILL.md` |
| auth, secrets, permissions | `~/ai-agent-skills-sync/skills/security-best-practices/SKILL.md` |
| HTTP APIs, REST endpoints | `~/ai-agent-skills-sync/skills/api-design-restful/SKILL.md` |
| TypeScript, type safety | `~/ai-agent-skills-sync/skills/typescript-best-practices/SKILL.md` |
| error handling, logging | `~/ai-agent-skills-sync/skills/error-handling-logging/SKILL.md` |
| user input, validation | `~/ai-agent-skills-sync/skills/validation-input-sanitization/SKILL.md` |
| tests, regressions | `~/ai-agent-skills-sync/skills/testing-patterns/SKILL.md` |
| database, data models | `~/ai-agent-skills-sync/skills/database-data-modeling/SKILL.md` |
| refactoring, design patterns | `~/ai-agent-skills-sync/skills/dry-solid-principles/SKILL.md` |
| performance, optimization | `~/ai-agent-skills-sync/skills/performance-optimization/SKILL.md` |
| git, commits, PRs | `~/ai-agent-skills-sync/skills/git-workflow/SKILL.md` |

When `~/.claude/CLAUDE.md` was generated by `./sync.sh to-local`, all skill content is already inlined — no separate read needed. Only use the table above if working from a fresh session where sync hasn't run.

## Suggesting learnings

At the end of any session where a reusable pattern emerged, suggest:

```bash
~/ai-agent-skills-sync/record-learning.sh generated "title" "details"
```

Use the same keyword taxonomy as `suggest_promotion` in `learningUiServer.py` to pick the right target skill.
